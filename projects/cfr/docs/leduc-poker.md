# Leduc poker

Kuhn poker の次に来る定番ベンチマーク。公開カード・2ラウンド・レイズと、本物のポーカーに
一歩近い要素が入るが、まだ厳密に解ける小ささ。Kuhn を解いた CFR を次に試す題材。

## ルール

- **2人対戦**
- **カードは6枚**：J・Q・K を **各2枚**（強さ J < Q < K）
- 両者 **1チップずつアンティ** → ポットは 2
- 各自に **private card を1枚** 配る（伏せ）
- **2ラウンド**のベッティング。間に **ボードカード（公開カード）を1枚** めくる

## 進行

```
1. 各自に private card を1枚配る（伏せ）
2. ラウンド1のベッティング
3. ボードカードを1枚公開（残り4枚から）  ← ゲーム途中の chance
4. ラウンド2のベッティング
5. ショーダウン（降りていなければ役比較）
```

## ベッティング（固定リミット）

- **ベット額：ラウンド1 = 2、ラウンド2 = 4**（後のラウンドで倍）
- 各ラウンドの行動：**check / bet(raise) / call / fold**
- **レイズ上限：各ラウンド2回まで**
- 例（ラウンド1）：P1 check → P2 bet(2) → P1 raise(2) → P2 call … のように上限まで

合法手は状況で変わる（facing bet では check 不可、上限到達で raise 不可）。これは Kuhn
（常に2択）と違う重要点。

## 役の強さ（ショーダウン）

1. **ペア**（自分の private card が公開ボードと**同じランク**）→ 最強
2. ペアが無ければ、**private card のランクが高い方が勝ち**
3. 同ランクでペアも無し → **引き分け（ポット折半）**

ボードは1枚なので、ペアを作れるのは最大1人（各ランク2枚、1枚はボードが取る）。

### 例

| P1 | P2 | ボード | 勝者 | 理由 |
|----|----|--------|------|------|
| K | Q | Q | **P2** | Q のペアが K(ハイカード)に勝つ |
| K | Q | K | **P1** | K のペア |
| K | Q | J | **P1** | ペア無し、K > Q |
| Q | Q | J | 引分 | 同ランク・ペア無し |

## 鍵となる概念：情報集合（information set）

Kuhn と同じく不完全情報（相手の private card が見えない）。Leduc の情報集合 ＝
**(自分の private card) + (公開ボード※ラウンド2のみ) + (見えている行動履歴)**。

ラウンドとボードが加わるぶん、情報集合の数は Kuhn（12個）より大幅に多いが、まだ有限・小規模で
厳密に解ける。

## Kuhn から増える要素（設計への含意）

Leduc は CFR エンジンに、Kuhn には無かった対応を要求する。

| 新要素 | エンジン/設計への影響 |
|--------|----------------------|
| **ゲーム途中の chance node**（ボード公開）| 初期配牌だけでなく**途中のチャンスノード**を扱う必要 |
| **合法手が状況で変わる** | concept に **`LegalActions()`** 相当が要る（今は全 `kNumActions` を常に探索）|
| **2ラウンド** | State が「ラウンド・ポット・直近のベット数・ボード」を持つ |
| **ペア判定** | `Utility()` が private と board でペアを見る |

OpenSpiel では `CurrentPlayer() == kChancePlayerId` ＋ `ChanceOutcomes()`、`LegalActions()`
で扱っている要素。Kuhn を壊さずこれらを足すのが Stage（Leduc 追加）の本質的な作業。

## GTO・検証

Leduc も**ナッシュ均衡（GTO）が計算済みの標準ベンチマーク**。最適戦略は混合戦略で、ブラフや
スロープレイが現れる。自作 CFR を Leduc で走らせ、**公開されている均衡値・戦略に収束するか**で
正しさを検証できる（Kuhn の −1/18 と同じ役割。[cfr.md](cfr.md) の検証方法を参照）。
